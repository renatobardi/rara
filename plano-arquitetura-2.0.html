De coreografia para um control plane orquestrado

O 2.0 dá ao rara um orquestrador explícito — o rara-core — sem perder o desacoplamento que torna o sistema robusto. Este documento é didático: explica cada peça, mostra os fluxos e o modelo de dados, e destaca o que já foi decidido e o que falta decidir.

1A virada

Hoje (1.0) o rara são cinco agentes Go (harvest, shelf, scribe, distill, feed) coordenados por coreografia: cada um roda no próprio cron e se comunica só pelas tabelas no Neon, via upserts idempotentes. Nenhum agente chama outro. Isso é uma força — idempotente, isolado, re-executável, imune a falha em cascata.

O 2.0 adiciona um control plane determinístico que decide o que roda em seguida, onde roda e se vale a pena fazer — enquanto os workers continuam tão desacoplados quanto hoje. O contrato entre control plane e workers continua sendo a tabela no banco, nunca uma chamada direta.

Decidido — Modelo de orquestração: control plane sobre coreografia (reconciler determinístico). Julgamento (LLM) vive só dentro dos portões, nunca no control flow.

2Os 4 conceitos que carregam tudo

É, na prática, "o LiteLLM, só que um nível abaixo": o mesmo padrão de router (custo/qualidade/fallback/constraints) aplicado a onde o trabalho roda.

🧩

Capability

Uma tarefa lógica com contrato fixo: coletar, transcrever, extrair, gate_barato, gate_rico, destilar.

⚙️

Provider

Implementação concreta de uma capability, com tags: runtime, activation, custo, qualidade, constraints. Adicionar worker = registrar provider.

🔀

Flow

Pipeline declarativo por raia: passos ordenados com options (ex.: gate:skip). Mudar pipeline = editar flow.

🎚️

Routing policy

Peso custo ⇄ qualidade + constraints duras + fallback ordenado, global ou por capability.

Decidido — Config é dado, não código, guardada em tabelas no Neon e exportável pro git. Worker novo ou pipeline re-cabeado não exige mudança no control plane nem redeploy.

3Control plane — o reconciler

O rara-core roda um reconciler always-on na VPC (o Mac dorme, o Cloud Run escala a zero; o loop de controle precisa ficar acordado). Ele é level-triggered, no sentido de controller do Kubernetes: observa o estado desejado (flows + items) vs o atual (item_steps + saúde dos providers) e age.

4Despacho — pull pro trabalho, ativação configurável

A entrega do trabalho é uniforme: o reconciler grava a atribuição e o worker puxa do Neon com SELECT … FOR UPDATE SKIP LOCKED (fila do próprio Postgres — sem broker, NAT-friendly). Só a ativação varia por provider.

AtivaçãoUsado porComo acorda
residentApp de tray no Mac, worker residente na VPCjá acordado, faz polling
on_demandJobs do Cloud Run (escala a zero)reconciler chama Cloud Run Jobs run; o job sobe, puxa e morre
Sequência de despacho
sequenceDiagram

  participant N as Neon (item_steps)
  participant W as Worker resident
  participant CR as Cloud Run Job
  R->>N: atribui(item, step, provider)
  alt provider resident
    W->>N: claim (FOR UPDATE SKIP LOCKED)
  else provider on_demand
    R->>CR: run (ativação)
    CR->>N: claim (FOR UPDATE SKIP LOCKED)
  end
  Note over W,CR: executa a capability
  W-->>N: grava resultado + status + heartbeat
  CR-->>N: grava resultado + status + heartbeat
  R->>N: observa e avança
Decidido — Entrega sempre por pull; ativação configurável por provider. Seleção de provider centralizada no reconciler, com timeout→reativar/fallback pelo heartbeat.

5Curadoria — dois portões cost-aware

O 1.0 destila tudo; o 2.0 seleciona. Um portão é só uma capability. São dois, posicionados por custo: gate_barato roda sobre metadata antes de pagar ASR; gate_rico roda sobre o texto completo antes de pagar a destilação.

As quatro camadas se compõem numa cascata barato→caro, pra que o LLM raramente rode:

Por que a quarentena existe — item largado de cara no portão 1 nunca é transcrito nem visto, então você nunca recebe feedback de um falso-negativo (cold-start). O estado defer guarda uma amostra revisável pra calibrar.

6O loop de aprendizado

Um loop fechado por um artefato legível por humanos — o interest_profile (temas, autores, anti-temas, pesos), vivo e versionado. Sem infra de treino, sem lock-in.

8Roteamento — uma capability, vários runtimes

O router ordena por custo ou qualidade e respeita constraints duras. O exemplo que isso compra: o áudio do YouTube exige IP residencial, então o router elimina Cloud Run/VPC e sobra o Mac — a divergência que hoje é manual vira policy.

11Postura anti-lock-in

Três camadas independentes, decididas separado, pra que nenhum fornecedor seja estrutural:

1 · Control plane

Você possui (Go fino). Se um dia precisar de durabilidade, troca por Restate/Temporal self-host — o contrato é a tabela, workers não mudam. Lock-in de framework, reversível.

2 · Modelos

Atrás do LiteLLM. Claude/Gemini/Groq viram engines trocáveis por config.

3 · Integração

Padrões abertos: MCP (tools, Linux Foundation) + A2A (agente-a-agente). ACP está deprecado.

Resultado — A Anthropic acaba sendo (a) um modelo atrás do LiteLLM e (b) um possível cliente MCP que dirige o rara. Os dois opcionais e trocáveis.
